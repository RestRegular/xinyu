# 心隅游戏后端提示词系统重构方案

## 一、现状分析

### 1.1 当前问题

经过对现有代码的全面审查，发现提示词系统存在以下核心问题：

**分散存放问题**

提示词内容散落在多个文件中，`aiService.js` 包含 GM 系统提示词、角色 AI 提示词和 UserAgent 提示词，`gmPipeline.js` 包含部分提示词构建逻辑，`game.js` 的 autofill 接口中也包含独立的提示词。这种分散导致修改一处提示词需要在多个文件中查找。

**职责重叠问题**

GM 系统提示词在一个函数中混合了多种职责：角色设定、世界观描述、玩家状态展示、工具使用指南、输出格式说明、行为规则等。这些内容本应分层管理，却被压缩在一个超长字符串中。

**硬编码问题**

世界观预设（奇幻、科幻、武侠等）被硬编码在 `aiService.js` 中，语气预设也被硬编码。每次新增世界观类型都需要修改核心代码文件。

**维护困难问题**

修改提示词需要阅读数百行的字符串拼接代码，难以快速定位目标内容。代码审查时难以区分逻辑变更和提示词文本变更。

### 1.2 重构目标

- 将提示词与代码逻辑分离，实现关注点分离
- 建立可扩展的预设系统，支持用户自定义世界观
- 实现提示词的热重载能力，提升开发效率
- 统一提示词管理入口，降低维护成本

## 二、重构方案设计

### 2.1 整体架构

新的提示词系统采用分层架构，共分为五个层级：

**第一层：模板存储层**

使用独立的 `.txt` 文件存储提示词模板，每个模板负责一个独立的语义单元。模板文件支持变量占位符，格式为 `{{变量名}}`，在运行时被实际值替换。

**第二层：注册管理层**

提示词注册表负责加载、缓存和管理所有模板文件。注册表在服务启动时扫描模板目录，将所有模板加载到内存中。在开发环境下，注册表会监听模板文件的变化，实现热重载。

**第三层：规则片段层**

将可复用的规则内容（如工具使用规则、角色交互规则、输出格式规则等）抽取为独立的 JavaScript 模块。这些规则片段既有结构化的数据结构，也可以包含格式化的文本内容。

**第四层：预设配置层**

世界观预设和语气预设从提示词中分离出来，成为独立的配置文件。每种世界观包含显示名称、图标、语气指南、世界规则、叙事技巧、物品命名风格等属性。

**第五层：构建器层**

每个 Agent（GM、角色AI、UserAgent、Autofill）拥有独立的提示词构建器。构建器从注册表获取模板、从预设获取配置、从规则片段获取规则，组合生成最终的提示词。

### 2.3 核心模块设计

**提示词注册表**

注册表是模板管理的核心组件。它负责以下功能：

- 启动时扫描 `templates/` 目录，加载所有 `.txt` 文件
- 将模板内容按文件名存储到内存 Map 中
- 提供 `get(name)` 方法获取原始模板
- 提供 `render(name, vars)` 方法进行变量替换渲染
- 提供 `compose(sections, vars)` 方法组合多个模板片段
- 在开发环境下监听模板文件变化，自动重新加载

变量替换支持嵌套对象访问，例如 `{{player.attributes.hp.current}}` 会被替换为对应的属性值。如果变量不存在，替换为空字符串。

**预设系统**

预设系统采用配置化设计，每种世界观是一个配置对象，包含以下字段：

- `id`：唯一标识符
- `displayName`：显示名称
- `icon`：图标字符
- `toneGuide`：语气指南文本
- `worldRules`：世界规则描述
- `narrativeTips`：叙事技巧建议
- `itemNamingStyle`：物品命名风格
- `sampleNpcs`：示例 NPC 列表（可选）

预设系统提供一个 `getGenrePreset(genre)` 函数，当传入的世界观类型不存在时，返回一个包含默认值的自定义预设对象，确保系统不会因预设缺失而崩溃。

**规则片段模块**

规则片段分为两类：

一类是结构化规则，以 JavaScript 对象形式导出，便于程序化处理。例如工具映射表可以用对象数组表示，方便后续扩展。

另一类是格式化文本规则，以多行字符串形式导出，直接嵌入到最终提示词中。这类规则保持原有的自然语言描述风格，便于 LLM 理解。

**构建器设计**

每个构建器遵循相同的设计模式：

1. 接收 `saveData` 和 `appConfig` 作为输入
2. 从 `saveData` 中提取需要的数据
3. 调用预设系统获取世界观配置
4. 调用注册表渲染各个模板层
5. 组合规则片段
6. 返回完整的提示词字符串

构建器内部不包含业务逻辑，只负责数据的提取、转换和组合。

### 2.4 模板文件设计

模板文件采用分层设计，每个模板文件对应提示词的一个语义层：

**基础层模板**（`gm_layer_base.txt`）

包含 GM 的角色设定，描述 GM 的身份和职责。这一层相对稳定，通常不需要频繁修改。

**世界观层模板**（`gm_layer_world.txt`）

包含世界类型、描述、叙事基调、世界规则、叙事技巧、物品命名风格等内容。这一层大量使用变量占位符，实际内容来自预设配置或用户输入。

**角色层模板**（`gm_layer_character.txt`）

包含玩家角色的名称、性别、年龄、身份、外貌、性格、背景故事、等级等属性信息。

**情境层模板**（`gm_layer_context.txt`）

包含当前位置、生命值、魔力值、属性值、金币、背包物品、状态效果、NPC 列表、重要角色列表等动态信息。

**规则层模板**（`gm_layer_rules.txt`）

包含叙事长度要求、全局指令、世界自定义指令等规则性内容。

模板文件中的变量占位符使用双大括号语法。对于条件性内容，模板引擎暂不支持复杂逻辑，条件判断应在构建器中完成，即构建器根据条件决定是否包含某个模板片段。

## 三、迁移计划

### 3.1 第一阶段：基础设施搭建

创建 `prompts/` 目录结构，实现注册表模块和预设系统模块。

注册表模块需要实现文件扫描、内容加载、变量替换、热重载等核心功能。预设系统需要将现有的五种世界观（奇幻、科幻、武侠、末日、现代）从代码中迁移到配置文件中。

此阶段不修改现有代码，只是新增模块，确保新模块可以独立运行。

### 3.2 第二阶段：模板提取

分析现有提示词内容，将其拆分为多个语义独立的片段，分别写入对应的模板文件。

GM 系统提示词拆分为五个模板层，角色 AI 提示词拆分为一个模板文件，UserAgent 提示词拆分为一个模板文件，Autofill 提示词拆分为一个模板文件。

在拆分过程中，需要识别出所有变量占位符，确保模板中的变量名与构建器传入的变量名一致。

### 3.3 第三阶段：构建器实现

实现四个构建器：`gmPrompt.js`、`characterPrompt.js`、`userAgentPrompt.js`、`autofillPrompt.js`。

每个构建器先实现与原有函数相同的接口，内部使用注册表和预设系统生成提示词。构建器完成后，编写单元测试验证生成的提示词与原有一致。

### 3.4 第四阶段：代码替换

修改 `aiService.js`，将原有的提示词构建函数替换为调用新构建器。修改 `gmPipeline.js` 中的导入路径。修改 `game.js` 中的 autofill 接口，使用新的构建器。

替换完成后，删除原有的提示词相关代码，避免冗余。

### 3.5 第五阶段：测试与验证

进行功能测试，验证游戏的各种场景下提示词功能正常。特别关注以下场景：

- 不同世界类型的切换（奇幻、科幻、武侠、末日、现代）
- 自定义世界类型的处理
- 重要角色的创建和交互
- 地点的创建和移动
- 物品的获取和使用

## 四、风险与应对

### 4.1 模板变量不一致风险

在拆分模板时，可能出现变量名拼写错误或大小写不一致的情况，导致运行时变量无法被正确替换。

**应对措施**：在构建器中添加变量校验逻辑，对必需的变量进行检查，缺失时输出警告日志。同时编写集成测试，覆盖所有模板的渲染场景。

### 4.2 热重载性能风险

在开发环境下启用热重载时，频繁的文件变化可能触发多次重载，影响开发体验。

**应对措施**：对文件变化事件进行防抖处理，在文件变化后等待 100 毫秒再执行重载。同时只在开发环境启用热重载，生产环境禁用。

### 4.3 兼容性风险

重构后生成的提示词可能与原有提示词存在细微差异，导致 AI 行为发生变化。

**应对措施**：在替换阶段保留原有函数，新增函数并行运行一段时间。可以通过配置开关控制使用新旧版本，便于 A/B 对比测试。

## 五、后续扩展方向

### 5.1 用户自定义预设

当前预设是内建的，未来可以支持用户创建和导入自定义世界观预设。用户可以通过界面编辑世界观配置，保存到数据库，系统启动时加载。

### 5.2 多语言支持

模板系统天然支持多语言。可以为每种语言创建独立的模板目录（如 `templates/zh/` 和 `templates/en/`），根据用户语言偏好选择对应的模板。

### 5.3 提示词版本管理

可以为模板文件增加版本号字段，在运行时检测模板版本，当模板更新时提示用户或自动迁移。这有助于在游戏进行中更新提示词而不破坏存档兼容性。

### 5.4 A/B 测试框架

基于模板系统可以构建提示词的 A/B 测试框架。将不同版本的模板放入不同的测试组，随机分配给用户，收集 AI 响应质量数据，持续优化提示词效果。

## 六、总结

本重构方案将提示词系统从代码中分离出来，建立了模板化、配置化、可扩展的新架构。重构后，提示词的修改不再需要修改核心代码，新增世界观类型只需添加配置文件，开发环境下支持热重载提升调试效率。

方案的实施分为五个阶段，预计总工时约为 8 小时。建议按阶段逐步推进，每个阶段完成后进行验证，确保系统稳定性。
